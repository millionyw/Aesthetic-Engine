# .ai_context

## 项目摘要
构建一个本地运行的基于 CLIP 的个人审美偏好引擎，用少量标注实现大规模图片筛选。

## 技术栈约定
- Python 3.10+ (小于 3.13)
- PyTorch（必须适配 MPS/Mac 或 CUDA/Nvidia 加速）
- HuggingFace Transformers（CLIP 模型）
- facenet-pytorch
- ultralytics
- faiss-cpu
- sqlite3（内置）
- streamlit-shortcuts
- 数据存储格式：Pickle/JSON
- 存储规范：所有模型（HuggingFace, PyTorch Hub, YOLO）必须强制下载并缓存在 ./data/models 目录下
- Linter: ruff check .
- Type Checker: 不使用 mypy，保留一定的灵活性

## 架构决策
1. **Linear Probing**: 选择 Linear Probing 而非 Fine-tuning：冻结大模型骨干，仅训练末端小模型，以降低样本需求、算力成本并提升隐私场景的可控性。
2. **混合特征**: 使用 Global CLIP + YOLO Person Crop CLIP + FaceNet 构建 2048 维混合特征，用于统一评分与可解释性分析。
3. **排序模型**: Pairwise 排名使用 Elo 作为排序基础。
4. **主键设计**：全系统（包括 SQLite、features.pkl、CSV）统一采用图片的**绝对路径 (Absolute Path)** 作为唯一主键 (Primary Key)，摒弃单纯的文件名，以彻底支持跨任意目录的灵活扫描与比对，并防止同名文件特征覆写。
5. **可解释性 (XAI)**：通过解析 Reward Model 的线性权重并与 CLIP 语义空间比对，实现对用户审美偏好的量化分析，支持从环境、人体、面部三个维度输出画像报告。
6. **视觉溯源 (Visual Tracing)**：基于架构解耦现状，采用 Occlusion Sensitivity (遮挡敏感度) 替代传统 Grad-CAM，实现黑盒模型的可视化解释。在 4GB 显存受限环境下，通过全局模型共享、推理模式 (`torch.inference_mode`) 及显式 VRAM 回收 (`torch.cuda.empty_cache`) 确保生成稳定性。
7. **显存管理策略**：针对受限显存 (4GB)，建立全局模型池 (`load_global_models`) 避免重复加载；在耗时推理任务（如 XAI）结束后强制执行 `gc.collect()`，防止显存泄露导致计算耗时指数级增长。

## Pairwise 采样策略 (Phase 12-12.5)
1. **Bridge (桥接采样)**：强制高分孤岛节点与最大连通簇(LCC)对比，打破图论孤岛。
2. **Exploration (探索采样)**：优先挑选对比次数极少的图片，建立基础连通性。
3. **Calibration (校准采样)**：高分(前10%)对战低分(后30%)，确立全局审美坐标尺。
4. **Refinement (精修采样)**：Ridge 预测分或 Elo 分极其接近的势均力敌对决，用于局部微调。

## 项目详细架构
- **核心目录结构**
  - /data/raw_images/（原图）
  - /data/models/（模型缓存）
    - /data/models/aesthetic_predictor.pkl（审美回归模型与标准化器）
  - /data/gallery.db（SQLite）
  - /data/features.pkl（多模态特征向量缓存）
  - /data/labels.csv（单图评分记录）
  - /data/pairwise_labels.csv（两两对比记录）
  - /src/（核心源码）
    - /src/pages/1_Versus_Arena.py（双向排序竞技场页面）
    - /src/visual_tracer.py（Phase 16: 视觉溯源核心模块）
- **核心模块职责**
  - ingest_images.py（ETL/入库）
  - feature_extractor.py（特征推理）
  - db.py（持久化）
  - app.py（UI/交互）
  - faiss_indexer.py（向量检索引擎，负责 4 种维度的近似最近邻搜索）
  - ranking_engine.py（Elo 排序、连通性/LCC、混合采样路由）
  - reward_engine.py（基于特征差分的泛化评分与不确定性采样）
  - export_engine.py（PDF 导出，保持长宽比，自适应布局与高 DPI）
  - visual_tracer.py（通过 Occlusion Sensitivity 生成审美热力图，支持 Batch 推理）
  - analyzer.py（审美偏好分析，负责解析 RM 权重、Concept Probing 语义标签探测及生成画像报告）
- **竞技场功能要点**
  - 右侧排行榜支持 Elo 与 Reward Model 切换（key: active_ranking_engine）
  - 过滤开关仅显示已比对次数 > 3（key: filter_min_matches）
  - 展示 σ（由对比次数推导）与 LCC 徽标；排行榜分页
  - PDF 导出随引擎切换输出对应排序结果
  - 点击左/右/跳过后延迟 rerun（request_rerun）避免未渲染控件状态丢失
  - **视觉溯源**: 在画像页点击“🔍溯源”触发 `st.dialog`，展示审美热力图。
- **数据库表结构**
  - images: path (PK), pred_score, human_score, timestamp

## 底层实现规范
- **特征向量结构（2048-dim）**
  - [0:768] Global Feature (CLIP-ViT 全局语义)。
  - [768:1536] Body Feature (YOLO 人体裁剪后经过 CLIP 提取)。
  - [1536:2048] Face Feature (FaceNet 提取的五官及面部比例特征)。
  - 权重解析原则：贡献度计算必须按维度数（768, 768, 512）进行均值归一化，以消除维度基数不均带来的偏差。
- **UI 状态管理原则 (State Management)**
  - **st.session_state**: 
    - 必须使用 `dialog_idx` 管理预览弹窗中的当前图片索引。
    - 必须使用 `st.cache_data` 处理缩略图生成以保证滚动性能。
    - 标注大厅与 Dialog 打分必须使用 `streamlit-shortcuts` 绑定 `1-5` 键位。
    - 竞技场：`active_ranking_engine`（引擎切换）、`filter_min_matches`（过滤）、`request_rerun`（延迟刷新）
    - 溯源弹窗：使用 `trace_dialog_open` 控制显示，`trace_pending_path` 存储待分析路径。
  - **状态流转**: 
    - 单图评分：更新数据库 -> 追加 labels.csv -> 触发静默重训 -> `st.rerun()`。
    - 两两对比：追加 pairwise_labels.csv -> 更新 Elo/连通性 -> 触发延迟 `request_rerun` -> `st.rerun()`。
    - 弹窗关闭必须手动清理索引状态，防止下次打开时位置错乱。

## 数据流程 (Data Workflow)
1. **入库**: 扫描本地 -> 提取三合一特征 -> Ridge 预测 -> 存入 SQLite。
2. **预览**: 按 `COALESCE(human, pred)` 降序读取 DB -> 渲染 4 列缩略图网格。
3. **闭环**: 用户在 Dialog 中使用键盘 (A/D/←/→) 翻页并打分 -> 增量训练模型 -> 实时更新预测分。
4. **溯源**: 点击按钮 -> 触发 Callback -> 更新 session_state -> 脚本顶部 Dialog 渲染 -> Occlusion 推理 -> 热力图展示。

## Memory Bank(经验沉淀)
- CLIP 模型加载应优先使用本地缓存路径；特征提取时需注意 Projection 层的维度匹配。
- 采用 RidgeCV 自动搜索正则化强度，并引入 StandardScaler 解决多模态特征尺度不一和分数压缩问题。
- Streamlit 性能优化：对于大图列表，必须使用内存缩略图字节流加载。
- 可解释性修正：多模态特征贡献度需消除维度基数偏差。
- **VRAM 优化**: 在 4GB 显存环境下，执行耗时深度学习任务后必须显式调用 `torch.cuda.empty_cache()` 和 `gc.collect()`，否则 VRAM 泄露会导致后续计算耗时呈指数级增长。
- **Streamlit Dialog**: 弹窗触发应使用“Callback + 顶部渲染”模式，避免在嵌套循环或复杂逻辑中直接调用导致的渲染失败。
